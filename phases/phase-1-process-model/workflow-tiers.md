# Workflow Tiers

## Complexity-Based Branching

모든 작업에 동일한 워크플로우를 적용하지 않는다.
복잡도에 따라 3단계로 분기하여 오버엔지니어링을 방지한다.

---

## Tier 정의

### Micro

**대상**: 단일 관심사의 소규모 변경

| 기준 | 범위 |
|------|------|
| 파일 수 | 1-2개 |
| 변경 유형 | 버그 수정, 단순 기능 추가, 설정 변경 |
| 의존성 영향 | 없음 또는 로컬 |
| 아키텍처 영향 | 없음 |

**워크플로우**:
```
작업 요청 → Developer 직접 실행 → 완료
```

- QR 생략
- 계획 단계 생략
- 문서화: 변경 사항이 기존 문서와 충돌하면 갱신

**예시**:
- 환경 변수 이름 변경
- 단일 함수의 버그 수정
- 의존성 버전 업데이트
- 설정 파일 수정

### Standard

**대상**: 다중 관심사의 중간 규모 변경

| 기준 | 범위 |
|------|------|
| 파일 수 | 3-10개 |
| 변경 유형 | 기능 구현, 리팩토링, 모듈 수정 |
| 의존성 영향 | 제한적 (같은 패키지 내) |
| 아키텍처 영향 | 경미 |

**워크플로우**:
```
작업 요청 → 간략 계획 → Developer 실행 → QR 1회 → 완료
```

- 간략 계획: 변경 파일 목록, 접근 방식, 주요 결정 1-2줄
- QR 1회: MUST + SHOULD 검사
- **QR 실행 시점**: 모든 코드 변경 완료 후, 커밋 전에 실행
- 문서화: 변경 범위에 해당하는 CLAUDE.md/README.md 갱신

> **실전 관찰 (M6)**: QR을 커밋 직전에 1회 실행한 결과, 에러 핸들링 누락과 테스트 갭을 발견하여 세션 내 수정 완료. Standard tier에서 QR 1회는 비용 대비 효과적.

**예시**:
- 새 API 엔드포인트 추가
- 기존 모듈 리팩토링
- 테스트 스위트 확장
- 에러 처리 개선

### Full

**대상**: 아키텍처 변경을 수반하는 대규모 변경

| 기준 | 범위 |
|------|------|
| 파일 수 | 10+개 |
| 변경 유형 | 아키텍처 변경, 시스템 통합, 대규모 기능 |
| 의존성 영향 | 넓음 (다중 패키지/모듈) |
| 아키텍처 영향 | 중대 |

**워크플로우**:
```
작업 요청
  → 상세 계획 (결정 로그, 대안 기각, 위험 분석)
  → QR 게이트 (계획 검증)
  → 마일스톤별 실행
    → Developer 실행
    → QR 게이트 (코드 검증)
    → 다음 마일스톤
  → 문서화
  → QR 게이트 (문서 검증)
  → 완료
```

- 상세 계획: 결정 로그, 기각된 대안, 위험 분석, 마일스톤 분해
- QR 게이트: 계획/코드/문서 각각에 대해 실행
- 마일스톤: 이전 마일스톤이 QR 통과해야 다음 시작
- 문서화: 아키텍처 결정, 불변 조건, 보이지 않는 지식 기록

**예시**:
- 인증 시스템 교체
- 데이터베이스 마이그레이션
- 마이크로서비스 분리
- 새 도메인 모듈 추가

## Complexity Assessment

### 자동 판정 기준

```
IF 파일 수 <= 2 AND 의존성 영향 없음:
  → Micro

ELSE IF 파일 수 <= 10 AND 아키텍처 영향 경미:
  → Standard

ELSE:
  → Full
```

### 수동 오버라이드

자동 판정이 부적절한 경우 수동으로 tier를 지정할 수 있다:

- "이 작업은 파일 2개지만 Standard로 진행해줘" → Standard 적용
- "이 작업은 파일 15개지만 변경이 기계적이므로 Micro로" → Micro 적용

### 도메인 프로필에 의한 조정

도메인 프로필의 `complexity_thresholds`가 기준을 오버라이드할 수 있다:

```yaml
# 금융 도메인: 더 보수적인 기준
complexity_thresholds:
  micro_max_files: 1      # 기본 2 → 1로 축소
  standard_max_files: 5   # 기본 10 → 5로 축소
```

## Tier 간 전환

### 업그레이드 (Micro → Standard, Standard → Full)

작업 중 복잡도가 예상보다 높아진 경우:

- 현재 단계의 작업을 완료 (중단하지 않음)
- 다음 작업부터 상위 tier 적용
- 기존 작업에 대해 QR을 소급 적용하지 않음

### 다운그레이드 (Full → Standard, Standard → Micro)

허용하지 않음. 작업 시작 시 결정된 tier를 유지.
예상보다 간단하더라도, 계획한 검증 단계를 생략하면 안 됨.

근거: 다운그레이드를 허용하면 "이건 간단하니까" 라는 판단 오류로
검증이 생략되는 패턴이 반복된다.
