# 구조적 롤링 요약 템플릿

## 개요

구조적 롤링 요약은 대화 컨텍스트가 잘릴 때(context truncation) 누적된 정보를 보존하기 위한 압축 메커니즘이다. Factory.ai의 구조적 롤링 요약 패턴에 기반하며, **강제 섹션**을 통해 "세부사항이 조용히 사라지는" 문제를 방지한다.

**핵심 원칙**:
- 컨텍스트가 잘릴 때만 동작 (불필요한 요약 방지)
- 잘린 구간만 요약하여 기존 롤링 요약에 병합
- 강제 섹션 구조를 통해 정보 범주별 보존 보장

---

## 강제 섹션 정의

모든 롤링 요약은 반드시 아래 5개 섹션을 포함해야 한다. 해당 섹션에 내용이 없더라도 섹션 헤더는 유지하고 "없음"으로 표기.

### 섹션 1: 수정된 파일 (Modified Files)

**토큰 예산**: 전체의 **25%**

**형식 규칙**:
- 파일 경로 + 변경 요약을 한 줄로 기록
- 형식: `- {path}: {요약} (M{N})`
- 마일스톤 번호를 괄호 안에 명시하여 "언제" 변경됐는지 추적
- 같은 파일의 여러 변경은 최신 것만 유지
- 삭제된 파일은 `[삭제됨]` 태그 표기

**정렬 기준**: 최근 변경순 (최신이 위)

### 섹션 2: 핵심 결정 (Key Decisions)

**토큰 예산**: 전체의 **30%**

**형식 규칙**:
- 형식: `- {선택} > {기각}: {근거 요약} (M{N})`
- 결정 근거를 반드시 포함 (근거 없는 결정은 무의미)
- **절대 삭제하지 않음** — 예산 초과 시 다른 섹션부터 줄임
- 철회된 결정은 `[철회됨, M{X}에서 대체]` 태그 추가

**정렬 기준**: 시간순 (오래된 것이 위, 결정 이력 추적)

### 섹션 3: 미해결 사항 (Unresolved Issues)

**토큰 예산**: 전체의 **20%**

**형식 규칙**:
- 형식: `- {이슈 설명} (M{N}에서 발생)`
- 해결된 항목은 제거 (해결 마일스톤은 결정 섹션에 기록)
- 긴급도 높은 항목에 `[긴급]` 태그 표기
- 3개 마일스톤 이상 미해결인 항목에 `[장기]` 태그 표기

**정렬 기준**: 긴급도순 → 발생 시간순

### 섹션 4: 아키텍처 변경 (Architecture Changes)

**토큰 예산**: 전체의 **15%**

**형식 규칙**:
- 새 모듈/레이어/서비스 추가, 의존성 구조 변경만 기록
- 형식: `- {변경 내용}: {영향 범위} (M{N})`
- 단순 파일 수정은 포함하지 않음 (그것은 "수정된 파일" 섹션)
- 의존성 추가/제거도 여기에 기록

**정렬 기준**: 시간순

### 섹션 5: 다음 단계 (Next Steps)

**토큰 예산**: 전체의 **10%**

**형식 규칙**:
- 형식: `- {작업 설명}`
- 완료된 항목은 제거
- 현재 진행 중인 마일스톤의 작업만 포함 (과거 마일스톤의 미완료 작업은 "미해결 사항"으로 승격)
- 최대 10개 항목

**정렬 기준**: 우선순위순

## 총 토큰 예산

| 상황 | 총 예산 | 근거 |
|------|---------|------|
| 기본 (128K 컨텍스트) | ~3,000 토큰 | 전체 컨텍스트의 ~2.3% |
| 대형 프로젝트 (20+ 마일스톤) | ~5,000 토큰 | 결정/파일 누적에 대응 |
| 소형 프로젝트 (5 이하 마일스톤) | ~1,500 토큰 | 오버헤드 최소화 |

예산 초과 시 가지치기 순서:
1. **다음 단계** — 가장 먼저 축소 (현재 마일스톤 것만 유지)
2. **수정된 파일** — 최근 3개 마일스톤 변경만 유지
3. **아키텍처 변경** — 핵심만 유지
4. **미해결 사항** — 축소하되 `[긴급]` 태그 항목은 유지
5. **핵심 결정** — **절대 축소하지 않음**

## 갱신 규칙

### 갱신 트리거 조건

롤링 요약은 다음 경우에만 갱신된다:

1. **컨텍스트 잘림 발생 시**: 대화 기록이 토큰 한도를 초과하여 잘릴 때, 잘린 구간의 정보를 롤링 요약에 병합
2. **마일스톤 완료 시**: 델타 엔트리가 생성되면 롤링 요약도 동기화
3. **명시적 요청 시**: 사용자가 요약 갱신을 요청한 경우

> 불필요한 갱신은 토큰 낭비. "변경이 있을 때만" 갱신.

### 갱신 알고리즘

```
function update_rolling_summary(existing_summary, truncated_segment, new_delta):
    // 1. 잘린 구간에서 섹션별 정보 추출
    extracted = extract_by_section(truncated_segment)

    // 2. 기존 요약과 병합 (merge-rules.md의 규칙 적용)
    merged = {
        modified_files: merge_files(existing_summary.modified_files, extracted.files),
        key_decisions: append(existing_summary.key_decisions, extracted.decisions),
        unresolved: merge_unresolved(existing_summary.unresolved, extracted.unresolved),
        architecture: append(existing_summary.architecture, extracted.architecture),
        next_steps: replace(extracted.next_steps)  // 최신 것으로 교체
    }

    // 3. 델타 엔트리가 있으면 추가 반영
    if new_delta:
        merged = apply_delta(merged, new_delta)

    // 4. 예산 체크 및 가지치기
    merged = prune_to_budget(merged, TOKEN_BUDGET)

    return format_as_markdown(merged)
```

### 재생성 트리거

전체 재생성(기존 요약 폐기 후 처음부터 생성)이 필요한 경우:

- 누적된 델타 로그와 롤링 요약 간 불일치 감지 시
- 롤링 요약의 정보 손실률이 20%를 초과할 때
- 프로젝트 구조의 대규모 변경 (디렉터리 재구성 등)

## 품질 검증 (정보 손실 감지)

### 검증 항목

| 검증 | 방법 | 실패 시 조치 |
|------|------|------------|
| 결정 완전성 | 델타 로그의 모든 결정이 요약에 존재하는지 확인 | 누락 결정 추가 |
| 미해결 사항 정합성 | 해결되지 않은 항목이 요약에서 사라지지 않았는지 확인 | 누락 항목 복원 |
| 파일 최신성 | 최근 3개 마일스톤의 변경 파일이 모두 포함되는지 확인 | 누락 파일 추가 |
| 아키텍처 연속성 | 추가된 모듈/서비스가 요약에 반영되는지 확인 | 누락 변경 추가 |
| 토큰 예산 | 총 토큰이 예산을 초과하지 않는지 확인 | 가지치기 실행 |

### 정보 손실률 계산

```
loss_rate = 1 - (요약에 포함된 핵심 항목 수 / 델타 로그의 전체 핵심 항목 수)

핵심 항목 정의:
- decisions: 모든 항목 (가중치 1.0)
- unresolved (미해결 상태): 모든 항목 (가중치 1.0)
- files_changed (최근 5개 마일스톤): 모든 항목 (가중치 0.8)
- lessons (최근 3개 마일스톤): 모든 항목 (가중치 0.5)
```

## 실전 예시: 마일스톤 5 시점의 롤링 요약

아래는 E-커머스 백엔드 프로젝트의 M5 완료 시점 롤링 요약이다.

```markdown
## 수정된 파일

- src/api/payments.py: 결제 API 엔드포인트, Stripe 웹훅 처리 (M5)
- src/services/stripe_client.py: Stripe SDK 래퍼, 멱등성 키 자동 생성 (M5)
- tests/test_payments.py: 결제 플로우 단위 테스트 12건 (M5)
- src/auth/refresh.py: 리프레시 토큰 발급/갱신 엔드포인트 (M4)
- src/middleware/rate_limiter.py: IP 기반 Rate limiting, Redis 사용 (M4)
- src/auth/jwt.py: JWT 토큰 발급/검증, RS256, 만료 1시간 (M3)
- src/middleware/auth.py: 인증 미들웨어, 데코레이터 방식 (M3)
- src/models/user.py: User 모델, role 필드 (admin/user/viewer) (M3, M2에서 생성)
- src/api/routes.py: API 라우트 정의, 버전 관리 /v1 (M2)
- src/config.py: 환경변수 기반 설정 관리 (M1)
- src/models/base.py: SQLAlchemy Base 모델 (M1)

## 핵심 결정

- FastAPI > Flask: 비동기 지원, 자동 OpenAPI 문서 (M1)
- PostgreSQL > SQLite: 프로덕션 대응, ACID 보장 (M1)
- SQLAlchemy > raw SQL: ORM 생산성, 마이그레이션 도구 (M1)
- bcrypt > argon2: 기존 의존성 호환, 성능 충분 (M2)
- JWT > 세션: 무상태 우선, 수평 확장 시 세션 스토어 불필요 (M3)
- RS256 > HS256: 비대칭 키로 서비스 간 검증, 마이크로서비스 대비 (M3)
- Redis > 인메모리: Rate limit 카운터, 서버 재시작 시 유지 (M4)
- Stripe > 자체 결제: PCI DSS 위임, MVP 속도 우선 (M5)

## 미해결 사항

- RBAC 세부 권한 모델 미정의 (M3에서 발생) [장기]
- 리프레시 토큰 탈취 시 무효화 전략 미정의 (M4에서 발생)
- 환불 정책 및 자동 환불 로직 미구현 (M5에서 발생) [긴급]

## 아키텍처 변경

- 프로젝트 초기 구조 생성: FastAPI + PostgreSQL + SQLAlchemy (M1)
- 인증 미들웨어 레이어 추가: 데코레이터 기반 라우트 보호 (M3)
- Redis 의존성 추가: Rate limiting, 향후 캐싱 활용 예정 (M4)
- Stripe 외부 서비스 연동: 웹훅 수신 엔드포인트, 멱등성 보장 (M5)

## 다음 단계

- 환불 API 구현 (POST /api/v1/payments/refund)
- 결제 실패 알림 시스템 연동
- RBAC 권한 매트릭스 설계 및 구현
- Stripe 웹훅 서명 검증 테스트 추가
- 리프레시 토큰 회전(rotation) 정책 구현
```
