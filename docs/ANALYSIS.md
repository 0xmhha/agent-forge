# 프로젝트 배경 분석

## 1. LLM 기반 개발의 현재 상태

### 1.1 관찰된 문제

LLM 기반 코드 개발은 단기적으로 생산성을 높이지만, 장기적으로 기술 부채를 가속화한다.

**LLM이 생성한 코드의 특성**:
- 자신이 만든 중복을 감지하지 못한다
- 파일 간 불일치를 인식하지 못한다
- God Function이 성장하는 것을 알아채지 못한다
- 명시하지 않은 것은 추론하지 않는다
- 컨텍스트 창이 커져도 중간 내용에 대한 주의력은 분산된다

**반복 루프 기반 도구(Ralph 등)의 한계**:
- 토큰 소비가 기하급수적으로 증가
- 반복할수록 작업 품질이 저하될 확률이 높아짐
- 비용 효율성과 품질 사이의 트레이드오프가 불명확

### 1.2 기존 접근의 분류

현재 LLM 기반 개발 도구/프레임워크는 다음 스펙트럼에 분포한다:

```
완전 자율 ◄────────────────────────────────────► 인간 주도

Aperant       ECC           SuperClaude    claude-config    Claude Code
(자동 실행)   (에이전트 팜)   (커맨드 체계)    (프로세스 강제)   (미니멀 CLI)
```

| 프로젝트 | 핵심 철학 | 문제 정의 | 강점 | 약점 |
|----------|----------|----------|------|------|
| **Aperant** | 완전 자동화 | "수동 코드 작업을 제거" | 병렬 실행, worktree 격리 | 잘못된 방향으로 빠르게 진행 가능 |
| **Everything Claude Code** | 에이전트 팜 | "토큰 낭비, 예측 가능한 실수" | 넓은 커버리지 (65 스킬) | 학습 곡선, 도구 과잉 |
| **SuperClaude** | 메타 프로그래밍 | "구조적 워크플로우 부재" | 30개 커맨드, 라이프사이클 전체 | 프로세스 강제력 부족 |
| **claude-config** | 프로세스 엔지니어링 | "LLM 코드가 빠르게 썩는다" | 이론적 기반, QR 게이트 | 복잡도, 검증 부재 |
| **Claude Code** | 미니멀 CLI | "AI 코딩을 접근 가능하게" | 단순함, 저마찰 | 구조 부재 |

### 1.3 독립적으로 수렴한 공통 원칙

서로 다른 철학에서 출발했지만, 성숙한 프로젝트일수록 다음 원칙에 수렴한다:

1. **계획과 실행의 분리** — 모든 프로젝트가 "먼저 계획, 그 다음 실행"에 동의
2. **에이전트 역할의 전문화** — "하나의 에이전트가 모든 것을 한다"는 접근은 없음
3. **리뷰 게이트** — 리뷰 없이 실행만 하는 성숙한 프로젝트는 없음
4. **컨텍스트 관리가 핵심 병목** — 모든 프로젝트가 "어떤 정보를 줄 것인가"를 문제로 다룸

## 2. Agent 기반 개발의 3단계 로드맵

### 2.1 1단계: 인간 개발팀 문화를 Agent로 재현

**현재 위치**: 대부분의 프로젝트가 여기에 있다.

인간 팀의 역할을 에이전트로 매핑:
- Architect = 시니어 엔지니어
- Developer = 구현 엔지니어
- Quality Reviewer = 코드 리뷰어
- Technical Writer = 테크니컬 라이터

이 단계의 핵심 과제:
- 인간 팀에서 암묵적인 역할 경계를 명시화
- 자연스러운 핸드오프를 형식적 프로토콜로 변환
- "내 범위가 아니다"를 에이전트에게 가르치기

이 단계의 한계: 인간 팀의 비효율성까지 그대로 복제한다.

### 2.2 2단계: Agent 특성을 반영한 최적화

**진입 조건**: 1단계가 안정적이고, 충분한 실험 데이터가 축적된 후.

Agent는 인간과 다르다:
- 컨텍스트 창은 유한하지만, 병렬성은 무한하다
- 기억은 없지만, 재현은 완벽하다
- 피로하지 않지만, 토큰 비용이 든다

최적화 방향:
- QR을 순차 반복 대신 병렬 다수결로
- 인간의 "경험"을 "패턴 라이브러리"로 대체
- 비용 곡선 기반 동적 모델 선택

### 2.3 3단계: Agent 전용 방식의 새로운 개념

**진입 조건**: 2단계에서 충분한 데이터와 실험이 축적된 후.

가능한 방향:
- 코드 대신 "의도 그래프"를 에이전트가 직접 조작
- 에이전트가 자신의 워크플로우를 결과 기반으로 자동 최적화
- 인간이 읽을 필요 없는 에이전트 간 최적화된 중간 표현

아직 아무도 도달하지 못한 단계. 수많은 실험 데이터가 필요하다.

## 3. claude-config 심층 분석

### 3.1 프로젝트 개요

- **저자**: Leon Mergen
- **규모**: 400개 파일, 12개 스킬, 5개 에이전트
- **핵심 철학**: LLM 코드의 기술 부채를 프로세스 엔지니어링으로 해결

### 3.2 아키텍처

#### 에이전트 (5개, 역할 엄격 분리)

| 에이전트 | 모델 | 역할 | 경계 |
|----------|------|------|------|
| Architect | Opus | 설계 | 코드 구현 안 함 |
| Developer | Sonnet | 구현 | 아키텍처 결정 안 함 |
| Debugger | Sonnet | 진단 | 버그 수정 안 함 |
| Quality Reviewer | Sonnet | 검증 | 코드 구현 안 함 |
| Technical Writer | Sonnet | 문서화 | 코드 구현 안 함 |

#### QR 3-Tier 규칙

| 규칙 | 심각도 | 대상 |
|------|--------|------|
| RULE 0 | MUST (절대) | 지식 보존, 프로덕션 안정성 |
| RULE 1 | SHOULD | 프로젝트 규약 준수 |
| RULE 2 | SHOULD/COULD | 구조 품질, 자동 수정 가능 항목 |

#### 반복별 디에스컬레이션

- iter 1-2: 모든 심각도 차단
- iter 3-4: MUST + SHOULD만
- iter 5+: MUST만

#### Temporal 규칙 ("시간 없는 현재")

코멘트에서 변경 이력을 제거하는 원칙:
- "Added mutex for thread safety" → "Mutex serializes concurrent access"
- "Replaces old approach" → "Thread-safe: each goroutine independent"

#### 컨벤션 계층

```
Tier 1: 명시적 사용자 지시
Tier 2: 프로젝트 문서 (CLAUDE.md, README.md)
Tier 3: 기본 규약 (structural.md)
```

#### Python 아키텍처

실행 엔진이 아닌, 가이던스 출력 시스템:
```
Python 스크립트 → 가이던스 출력 → LLM 판단 → 다음 스크립트 호출 → 반복
```

### 3.3 강점

1. **문제 정의의 정확성** — 올바른 문제를 풀고 있다
2. **역할 분리의 엄격함** — 역할 혼동에 의한 품질 저하 방지
3. **QR Gate 디에스컬레이션** — 완벽주의와 실용주의의 균형
4. **Temporal 규칙** — 문서 품질에 직접적 영향
5. **자기 참조적 개선** — 연구 기반 시스템

### 3.4 우려 사항

1. **복잡도** — 시스템 자체를 이해하는 학습 비용이 높다
2. **검증 부재** — formal benchmark 없이 관찰 기반만
3. **오버엔지니어링** — 간단한 작업에 과도한 워크플로우
4. **Claude Code 종속성** — 이식성이 낮다
5. **비용 효율성 미검증** — QR 반복 비용 대비 효과 불분명

### 3.5 추출 대상

파일/코드가 아닌, 원칙과 규약만 추출:
- QR 3-Tier 규칙 → code-reviewer 에이전트 통합
- Temporal 규칙 → 문서화 규약 추가
- 계획-실행 분리 강제 → 워크플로우 게이트 추가
- 컨벤션 계층 → 도메인 프로필 포함한 4-tier로 확장

## 4. 도메인 특화 Agent 가이드라인의 필요성

### 4.1 General Agent의 한계

범용 에이전트는 모든 도메인의 "평균"을 학습한다:
- 금융: 규정 준수 > 속도
- 게임: 성능 > 가독성
- 의료: 안전성 > 모든 것
- 스타트업 MVP: 속도 > 품질

같은 "코드 리뷰"라도 도메인에 따라 우선순위가 완전히 다르다.

### 4.2 설계 방향

도메인 프로필(YAML)로 에이전트 행동을 프로젝트별 커스터마이징:

```
도메인 프로필
├── 우선순위 매트릭스 (안전성, 성능, 규정준수, 속도, 유지보수성)
├── 규약 오버라이드 (테스트 전략, 리뷰 깊이, 문서화 수준)
├── 도메인 지식 (용어 사전, 금지 패턴, 필수 패턴)
└── 워크플로우 커스터마이징 (QR 설정, 에이전트 구성, 자동화 수준)
```

컨벤션 계층에 Tier 1.5로 삽입:
```
Tier 1:   명시적 사용자 지시
Tier 1.5: 도메인 프로필
Tier 2:   프로젝트 문서
Tier 3:   기본 규약
```

## 5. 토큰 효율적 누적 컨텍스트 관리

### 5.1 핵심 과제

코드가 누적될수록 검토량이 증가하지만, 토큰 예산은 고정이다.

3개의 하위 문제:
- **압축**: 같은 정보를 더 적은 토큰으로
- **선택**: 현재 작업에 필요한 것만
- **지속성**: 빠진 정보를 나중에 복구 가능하게

### 5.2 관련 연구

| 기술 | 출처 | 효과 |
|------|------|------|
| ACE 델타 엔트리 | arXiv:2510.04618 | 에이전트 작업 +10.6%, 결정론적 병합 |
| Meta-RAG | arXiv:2508.02611 | 79.8% 코드베이스 압축 |
| Merkle Tree | Cursor | O(log n) 변경 감지 |
| ACON 관찰 압축 | SWE-bench | 26-54% 토큰 절감 |
| Factory.ai 롤링 요약 | 36K 메시지 | 3.44 → 3.70 품질 향상 |
| Code-Craft 호출 그래프 | arXiv:2504.08975 | 82% 검색 정확도 향상 |

상세 기술 레퍼런스: [references/context-management.md](references/context-management.md)

### 5.3 제안 아키텍처

5개 레이어 구성:

```
Layer 5: 세션 관리 (롤링 요약 + 관찰 압축)
Layer 4: 컨텍스트 구성 (ACE 델타 + 결정론적 병합)
Layer 3: 검색 (Meta-RAG + LSH 중복 제거)
Layer 2: 코드 이해 (AST 청킹 + 호출 그래프)
Layer 1: 변경 감지 (Merkle 트리)
```

핵심 원칙: 비싼 연산(LLM 요약)은 오프라인에서 변경된 것만.
저렴한 연산(해시 비교, 규칙 병합)은 매 쿼리마다.
