# Agent Forge v2 — Vision Document

> 이 문서는 2026-03-26 세션에서의 대화를 기반으로 작성되었습니다.
> 아이디어의 원문, 분석, 설계 결정사항을 보존하여 추후 세션에서 맥락을 유지합니다.

## 1. 배경과 동기

### 1.1 문제 인식

현재 AI 코딩 도구들(Claude Code, Codex, Gemini)은 모두 **CLI 기반**으로 동작한다.
이는 두 가지 사용자 그룹에 대해 각각 다른 문제를 만든다:

- **CLI에 익숙하지 않은 일반 유저**: 진입 장벽이 높다
- **CLI에 익숙한 전문 개발자**: 한 번에 하나의 세션만 관리할 수 있어 병렬 작업이 어렵다

### 1.2 영감의 원천

두 오픈소스 프로젝트에서 영감을 받았다:

| 프로젝트 | 라이선스 | 영감 | 적용 |
|----------|---------|------|------|
| **VSCodium** | MIT | VS Code에서 MS 독점 부분을 제거한 순수 오픈소스 에디터 | CLI에 익숙하지 않은 유저에게 GUI 환경을 제공 |
| **tmux** | ISC | 터미널 멀티플렉서. 여러 세션을 백그라운드에서 관리 | 전문 개발자가 여러 AI 세션을 병렬로 디스패치/관리 |

### 1.3 핵심 관찰

> "CLI에서 동작하는 AI 도구들은 프로세스로 취급할 수 있다.
> 작업이 인터랙션을 필요로 하지 않고, 무엇을 수행해야 할지 명확하다면,
> tmux를 통해 작업을 수행하고 정리하고 끝나도록 수행이 가능해진다."

이것은 AI를 **도구가 아닌 워커(worker)로 취급**하는 패러다임 전환이다.

```
기존 패러다임:  개발자 → [AI 도구] → 코드
새 패러다임:    개발자 → [오케스트레이터] → AI Worker 1 → 작업 완료
                                         → AI Worker 2 → 작업 완료
                                         → AI Worker 3 → 작업 완료
```

## 2. 비전 정의

### 2.1 한 줄 요약

**웹 기반 AI 에이전트 오케스트레이션 플랫폼. 여러 AI CLI 도구를 세션으로 관리하고, 전문화된 스킬/플러그인과 조합하여 복잡한 작업을 자동 수행한다.**

### 2.2 브라우저 메타포

GUI 앱의 구조는 **웹 브라우저**와 동일한 메타포를 따른다:

```
브라우저                          Agent Forge
──────────                        ───────────
브라우저 앱 (Chrome)              → Agent Forge GUI Shell
탭                                → Agent Session (Claude/Codex/Gemini 각각)
URL 입력                          → 작업 지시 (프롬프트)
렌더링 엔진 (V8 + Blink)          → Agent Runtime (프로세스 관리 + 샌드박스)
Extension                        → Skills / Plugins / MCPs
Extension Store                  → Tool Registry (스코어링 포함)
DevTools                         → 세션 모니터링, 로그, 메트릭
북마크/히스토리                    → 작업 기록, delta-log, 그래프 메모리
탭 그룹                           → 프로젝트/워크스페이스
백그라운드 탭                      → 무인 실행 에이전트 (tmux 영감)
```

- 탭을 열면 새로운 AI 세션이 생성됨 (Claude, Codex, Gemini 중 선택)
- Extension(스킬/플러그인/MCP)은 모든 탭에 적용되거나 특정 조건에서만 동작
- 각 탭은 독립된 터미널 에뮬레이터로 AI CLI를 실행

### 2.3 작업의 세분화와 전문 에이전트

현실의 작업을 적분하듯 세분화하면, 각각의 atomic한 조각은 서로 다른 전문화된 스킬/플러그인을 필요로 한다:

| 작업 영역 | 세부 도구/서비스 |
|-----------|----------------|
| 자료 조사 | 일반 웹서치, SNS 서치, 코드 서치 |
| 업무 프로세스 | Jira, Slack, Confluence |
| 코드 관리 | GitHub |
| 코드 구현 | 언어별 컨벤션, 프로젝트별 패턴 |
| 테스트 | 프로젝트별 서로 다른 테스트 환경 |
| 문서화 | 기술 문서, API 문서, 사용자 가이드 |
| 디자인 | 이미지 생성, 영상 제작 |

> "이것을 하나의 agent로 처리할 수 있는 수준이 아니다.
> 각각에 대해 매우 상세화된 에이전트와, 에이전트 간의 작업 분배 처리를 통해
> 자료 서치, 설계, 구현, 테스트, 이슈 생성, 또 다른 작업 진행 등의
> 사이클이 진행될 수 있는 구조가 필요하다."

### 2.4 Tool Registry와 스코어링

- 모든 것을 직접 만들 수 없고, 더 좋은 스킬과 툴이 계속 생성되고 있음
- 외부 도구를 **찾고, 사용하고, 장단점을 검토하고, 스코어링**하여 관리
- 어떤 상황에 어떤 도구가 더 적합한지를 시스템이 알고 있어야 함
- 이를 통해 유저 요구사항이 들어왔을 때:
  1. 더 자세한 요구사항을 구체화할 수 있는 질문을 생성
  2. 적합한 도구를 자동 선택
  3. 작업을 분할하고 정리

### 2.5 샌드박스 보안 전략

도커 컨테이너만을 의미하지 않는 **다층 격리 전략**:

```
Level 1: 디렉토리 격리
  └── 작업용 새 폴더 생성 → 그 내부에서만 에이전트 실행
  └── 상위 디렉토리 접근 불가

Level 2: 설정 기반 제한 (핵심)
  └── claude: settings.local.json → allow/deny 규칙
  └── codex: sandbox 모드 (기본 내장)
  └── gemini: sandbox 설정
  └── 에이전트 실행 전에 설정 파일을 통해 허용 범위를 명확하게 관리

Level 3: OS 수준 격리 (선택)
  └── Docker container / nsjail / chroot
  └── 네트워크 정책 (허용된 API만 접근)

Level 4: 통신 제한
  └── 에이전트 간 통신은 Forge 서버를 통해서만
  └── Tool Registry에서 스코어가 낮은 도구는 자동 차단
  └── 알 수 없는 존재와의 소통 제한
```

> "샌드박스 환경이면, 내부에서 어떤 작업을 하더라도 현재 시스템을 건드리지 못하도록
> 함으로써 문제를 줄일 수 있다."

### 2.6 장기 비전: Agentic OS

현재 만들려고 하는 도구의 최종 진화 방향:

1. **도구 생성 수단**: 현업의 여러 부분에서 에이전트화를 돕기 위한 다양한 도구를 생성
2. **프로그램 합성**: 에이전트들의 조합으로 또 다른 프로그램을 생성
3. **자동화 시스템 구축**: 기업이나 정부 등 디지털 시스템의 자동화 도구

자기 진화 메커니즘 (듀얼 부트로더 패턴):

```
[1st Bootloader] → [OS v1 실행]
                      │
                      ├── 문제 감지
                      ├── 수정 코드 생성
                      ├── 빌드 → 약속된 위치에 바이너리 저장
                      └── 재부팅 트리거
                           │
[2nd Bootloader] → [OS v2 실행 (수정된 버전)]
                      │
                      └── 다시 문제 감지 시 → 1st Bootloader로 복귀
```

> Layer 1-4가 충분히 성숙한 후 자연스럽게 도달하는 상태.
> 브라우저도 처음엔 HTML 뷰어였고, 지금은 사실상 OS이다. 같은 경로.

### 3.3 기술 스택

| 레이어 | 기술 | 선택 이유 |

### 3.4 프로세스 관리 방식

```
[Agent Forge Server]
       │
       ├── node-pty ──→ claude --dangerously-skip-permissions
       ├── node-pty ──→ codex --full-auto
       ├── node-pty ──→ gemini
       │
       │   각 프로세스는:
       │   ├── 독립된 작업 디렉토리 (샌드박스)
       │   ├── 독립된 설정 파일 (e.g., .claude/settings.local.json)
       │   ├── stdin/stdout → WebSocket → 브라우저 탭
       │   └── 상태 모니터링 (alive/busy/done/error)
```

## 4. 기존 agent-forge v1과의 관계

현재 agent-forge v1은 **"하나의 에이전트를 잘 쓰는 방법론"**이다.
비전 v2는 **"여러 에이전트를 조합해서 작업을 자동 수행하는 플랫폼"**이다.

v1은 폐기가 아니라 **v2의 첫 번째 built-in 콘텐츠**가 된다:

```
agent-forge v1 (현재)          agent-forge v2 (비전)
─────────────────────          ─────────────────────
phases/                   →    docs/methodology/ (참조 문서로 보존)
dist/skills/              →    registry/built-in/ (기본 제공 스킬)
tools/workspace-mcp/      →    registry/mcps/workspace/ (레지스트리에 등록)
tools/token-monitor-mcp/  →    core/budget/ (핵심 기능으로 승격)
```

기존 도구들과의 비교:

| 기존 도구 | 한계 | Agent Forge |
|----------|------|-------------|

| 개별 AI API 래퍼 | API만 호출 | CLI의 풀 기능 (파일 편집, git 등) 활용 |

## 6. MVP 정의

### 6.1 MVP 범위

**MVP = "하나의 작업을 웹에서 AI에게 시키고, 결과를 받아보는 것"**

```
MVP 범위 (Phase 1)
──────────────────
포함:
  ✅ 웹 브라우저에서 접속
  ✅ Monaco Editor로 코드 보기/편집
  ✅ 터미널 탭에서 Claude Code 실행 (1개 에이전트)
  ✅ 작업 디렉토리 격리 (폴더 기반 샌드박스)
  ✅ 작업 결과 (코드 변경) 확인
  ✅ 세션 상태 모니터링 (진행중/완료/에러)

미포함 (후속 Phase):
  ❌ 멀티 모델 (Codex, Gemini)
  ❌ Tool Registry / 스코어링
  ❌ 자동 작업 분해
  ❌ 그래프 DB / 그래프 메모리
  ❌ Extension 시스템
```

### 6.2 MVP 검증 가설

> "AI CLI 도구를 웹에서 프로세스로 관리하는 것이 동작하고, 가치가 있는가?"

이것이 검증되면 나머지는 확장이다.

## 7. 로드맵

```
Phase 1a: GUI Shell
  웹 앱 + Monaco + xterm.js + WebSocket
  Claude Code 하나를 웹에서 실행/관리

Phase 1b: Agent Runtime
  프로세스 스폰/관리, 디렉토리 격리
  설정 기반 샌드박스, 세션 상태 모니터링

Phase 2: 멀티 에이전트 + Tool Registry
  Codex, Gemini 추가
  스킬/플러그인 등록, 스코어링

Phase 3: Orchestrator
  작업 자동 분해, 에이전트 라우팅
  에이전트 간 통신, 결과 취합
  그래프 메모리

Phase 4: 생태계
  Extension marketplace
  커뮤니티 도구 공유
  자동 진화 메커니즘 (Agentic OS)
```

## 8. 위험 분석

| 위험 | 심각도 | 대응 |
|------|:------:|------|
| CLI 도구 업데이트로 인터페이스 변경 | 높음 | 어댑터 패턴으로 CLI별 래퍼 분리 |
| 웹에서 pty 프로세스 관리 보안 | 높음 | 인증 + 세션별 격리 + rate limiting |
| 에이전트 비용 폭발 (토큰 사용량) | 중간 | 토큰 버짓 시스템 (기존 token-monitor-mcp 활용) |
| 여러 CLI의 출력 포맷 파싱 | 중간 | 구조화된 출력 모드 우선 사용 (--json 등) |
| 범위 비대화 | 높음 | MVP 엄격 준수, Phase별 검증 후 진행 |

| 구성 요소 | 가능성 | 근거 |

## 10. 해결하려는 메타 문제

이 프로젝트 자체가 해결하려는 또 하나의 문제:

> "아이디어를 텍스트로 작성하면 매번 그때그때 생성되지만,
> 다시 작성하기 어렵고, 세션이 초기화되면 이전 대화가 남아있지 않다.
> 열심히 작성한 내용이 휘발된다."

이것은 Agent Forge가 해결해야 할 핵심 문제 중 하나이기도 하다:
- 모든 작업 기록의 구조적 보존
- 세션 간 컨텍스트 연속성
- 아이디어 → 구체화 → 구현의 전 과정 추적

- VS Code 다운로드 제품: Microsoft 독점 라이선스 (텔레메트리, 마켓플레이스 포함)
- VSCodium: MS 독점 부분 제거한 커뮤니티 빌드

### B. 독립 도구 방향 결정

> "독립도구면 더 좋을것 같아. web 기반으로 어디서든 사용할수 있을것 같아서"

### C. VSCodium + tmux 영감 설명

> "vscodium 은 cli에 익숙하지 않은 일반 유저들의 사용성을 올려주는 gui 환경을 제공해줄수 있고,
> tmux 는 cli에 익숙한 전문 개발자들이 한번에 여러 세션의 백그라운드에서 작업등을 지시하고 관리할수 있도록 도와줄수 있어.
> claude code, codex, gemini 모두 cli에서 동작해.
> 만약 작업이 어떠한 인터랙션을 필요로 하지 않는다면, 그리고 그 작업이 무엇을 수행해야할지 명확하다면,
> tmux를 통해 작업을 수행하고 정리하고 끝나도록 수행이 가능해진다는 얘기가 돼."

### D. 전체 비전 서술

> "브라우저 앱에서 tab을 누르면 새창이 열리고, 브라우저의 extension은 모든 tab에서 적용될수 있거나,
> 특정 url에서만 동작할수 있거나 등이 설정된 독립된 앱인데, 분리되어 있잖아?
> 브라우저앱을 gui 앱으로 두고, tab을 claude, codex, gemini cli를 수행한 터미널 에뮬레이터라고 하면,
> extension은 skill, plugin, mcp 등을 생각해볼수 있을것 같아."

> "작업하는 과정을 적분하듯 세분화하면 매우 상세하게 쪼개지고,
> 각각의 atomic한 조각들은 서로다른 전문화된 skill이나 plugin을 필요로 한다는것을 깨달았어."

> "이 모든것을 잘 설계된 에이전트와 스킬, 그리고 업무 환경에 대한 이해를 갖고 있는 에이전트,
> 방대한 데이터를 관리하는 그래프형 db, 그래프메모리 등을 조합한 환경에서
> claude, codex, gemini와 같은 제너럴 에이전트를 통해 이 모든 작업이 가능하도록 하고 싶어."

> "툴을 관리하는 것에서 취약한 툴들은 스코어를 낮춰서 사용하지 못하도록 해두면 되거든.
> 샌드박스 환경이면, 내부에서 어떤 작업을 하더라도 현재 시스템을 건드리지 못하도록 함으로써 문제를 줄일수 있다고 생각해."

> "이 툴의 자동화와 고도화가 극에 달할정도로 진화하면, 이것이 하나의 agentic os로도 가능해질것 같거든."

> "현재 만들려고 하는 툴은 현업의 여러 부분에서의 에이전트화를 돕기위한 다양한 툴을 생성할수 있는 수단이 되어야하고,
> 에이전트들의 조합으로 또다른 프로그램을 생성할수 있어야 하며,
> 그 프로그램들과 에이전트들을 통해서, 기업이나 정부 등 디지털 시스템에 있어서는
> 자동화될수 있는 시스템을 만드는것의 도구가 되어야해.
> 따라서 이거의 이름이 agent-forge가 되는거야."
