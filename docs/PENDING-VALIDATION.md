## Phase 2: Domain Profiles

| ID | 항목 | 검증 방법 | 선행 조건 |
|----|------|----------|----------|
| P2-V1 | 프로필 교체가 에이전트 재설정 없이 가능 | 동일 세션에서 fintech → startup-mvp 프로필 교체 후, QR 결과가 즉시 변경되는지 확인 | 실제 프로젝트에 도메인 프로필 적용 |

3. 동일 코드에 대해 리뷰 재실행 → MUST 발견이 줄어드는지 확인
4. 에이전트 재시작 없이 프로필 변경이 반영되면 PASS

## Phase 3: Context Management

| ID | 항목 | 검증 방법 | 선행 조건 |
|----|------|----------|----------|
| P3-V1 | 10+ 마일스톤 작업에서 토큰 소비 측정 | 델타 로그 적용 프로젝트에서 마일스톤별 토큰 소비 기록, 선형 증가 여부 확인 | Phase 4 측정 파이프라인 + 실제 10+ 마일스톤 프로젝트 |
| P3-V2 | 누적 컨텍스트 유무에 따른 품질 차이 관찰 | 동일 작업을 (a) 델타 로그 없이 (b) 델타 로그 있이 수행하여 재작업 횟수 비교 | 동일 난이도의 비교 가능한 작업 2건 |
| P3-V3 | 델타 로그에서 정보 손실 없음 확인 | 마일스톤 N에서 마일스톤 1-3의 결정을 참조해야 하는 상황 발생 시, 델타 로그만으로 해당 결정을 복구할 수 있는지 확인 | 5+ 마일스톤 프로젝트에서 델타 로그 사용 |

**P3-V1 검증 시나리오**:
1. Full 워크플로우 프로젝트 (10+ 마일스톤) 선정
2. 각 마일스톤 시작/완료 시 토큰 소비 기록
3. 기대: 마일스톤별 토큰이 일정 범위 내 유지 (선형 증가 아님)
4. 토큰/마일스톤 비율이 마일스톤 1 대비 2배 이내면 PASS

**P3-V2 검증 시나리오**:
1. 유사 규모의 작업 2건 준비
2. 작업 A: 델타 로그 없이 순수 컨텍스트로 진행
3. 작업 B: 델타 로그 + 롤링 요약 적용
4. 재작업 횟수, QR 반복 횟수, 최종 품질 비교
5. 작업 B의 재작업이 30% 이상 적으면 PASS

**P3-V3 검증 시나리오**:
1. 마일스톤 5+ 시점에서 의도적으로 마일스톤 2의 결정을 참조하는 작업 수행
2. 컨텍스트를 /clear한 후, 델타 로그만 주입
3. LLM이 해당 결정을 정확히 참조하면 PASS
4. 참조 실패 시: 델타 엔트리의 decisions 필드 보강 필요

## 검증 우선순위

```
P2-V1 (프로필 교체)     → Phase 4 없이도 즉시 검증 가능
P3-V3 (정보 손실)       → 5+ 마일스톤 프로젝트에서 검증
P3-V1 (토큰 소비)       → Phase 4 측정 파이프라인 필요
P3-V2 (품질 차이)       → Phase 4 + 비교 가능한 작업 2건 필요
```
